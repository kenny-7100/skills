# 对比 Main 做 CR

用于 review 当前分支相对最新 `main` 的改动。

## 流程

1. 确保当前分支干净：
   - 所有修改都必须已经 commit。
   - 如果当前分支不干净，先停止并提醒用户。

2. 确保 `main` 是最新的，并且和远程一致：
   - 需要根据仓库远端关系判断应该使用哪个远端的 `main`。
   - 如果是 fork PR 开发场景，优先考虑 `upstream/main`。
   - 如果不是 fork 场景，通常使用项目主远端的 `main`。
   - 如果本地 `main` 和目标远端 `main` 不一致，先保持一致。
   - 对比时优先使用目标远端的 `main`。

3. 对比当前分支和最新 `main`，然后做 CR。


## CR问题列表输出格式

输出必须是 Markdown，并使用固定列表结构：

```md
## CR 问题列表

1. **级别**：严重 / 中等 / 轻微
   **位置**：文件路径:行号
   **问题**：说明具体问题。
   **影响**：说明可能造成的影响。
   **复现路径**：说明如何触发或验证这个问题。
   **建议**：说明建议如何修改。
   **一句话总结**：用通俗、容易理解的方式解释这个问题。
```

如果没有发现问题，输出：

```md
## CR 问题列表

未发现明显问题。
```

## 注意

- 避免：自动部署、push、merge、rebase、reset。
- 避免给出非本批次修改引入的问题，避免给出历史问题。
- 确保影响已有功能的问题都被识别出来。
